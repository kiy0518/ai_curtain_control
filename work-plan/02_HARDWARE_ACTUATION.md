# 02. 커튼 물리 제어 (Motor Controller — Serial)

> 상태: **보류(추후 진행)** · 외부 의존: **모터 컨트롤러 측 준비 후** 착수
> 현재는 인터페이스 자리/프로토콜 **초안**만 확보. 대시보드는 제어 UI를 **placeholder**로 먼저 만들고(`04_*`), 컨트롤러 준비 시 이 계층을 붙인다.

### 확정된 전제 (사용자 확인)
- 모터 컨트롤러 **기존 명령 규격 없음** → 프로토콜은 추후 양측 정의 (아래는 초안 제안).
- **모터 속도 고정** → 속도 설정 명령 **불필요**(프로토콜에서 제외).
- **위치 피드백 미정** → 컨트롤러 연동 시 결정(우선 개/폐/정지만 가정).
- 연결: **UART** (`/dev/ttyS?`).

제스처/스케줄로 결정된 동작을 **모터 컨트롤러**로 전달해 실제 커튼을 구동하는 계층.
연결은 **UART 시리얼** (보드 GPIO 직접 구동 X — 컨트롤러가 모터 전력단 담당).

```
[제스처/스케줄/대시보드] → control 로직 → 시리얼 드라이버 ──(UART)──► [모터 컨트롤러] → 모터/커튼
                                            ▲ 상태/ACK ◄───────────────┘
```

## 목표
- 열림/닫힘/정지 명령을 UART 시리얼로 송신 (속도 고정, 위치는 피드백 지원 시).
- 컨트롤러 상태(이동상태·끝단·오류) 수신·반영.
- 통신 프로토콜을 명확히 정의(양측 합의) + 재연결/타임아웃/오류 처리.

## ⚠️ 보드측 전제 (착수 전 필수 확인 — 검증으로 발견)
- **dialout 권한**: 현재 `radxa` 계정은 `dialout` 그룹에 없음 → `/dev/ttyS*` 열기 시 PermissionError.
  → `sudo usermod -aG dialout radxa` (또는 systemd 유닛에 `SupplementaryGroups=dialout`). `install.sh`에 포함.
- **UART 활성화**: 현재 `/dev/ttyS*` 노드 없음. RK3576은 UART가 **디바이스트리 오버레이**로 활성화돼야 생성됨.
  → `rsetup`/오버레이로 해당 UART 활성화 + 물리핀 매핑 확인 + **루프백(TX-RX 점퍼)** 으로 검증 후 컨트롤러 연결.

## 시리얼 연결 파라미터 (config.env)
| 항목 | 기본값(예) | 비고 |
|------|-----------|------|
| `MOTOR_SERIAL_PORT` | `/dev/ttyS3` | 보드 **UART** (핀/포트번호 확인 필요) |
| `MOTOR_BAUDRATE` | `115200` | 8N1 |
| `MOTOR_TIMEOUT` | `0.3` s | 읽기 타임아웃 |
| `MOTOR_ACK_TIMEOUT` | `1.0` s | 명령 ACK 대기 |

## 통신 프로토콜 (제안 — v1)

ASCII 라인 기반, 줄 끝 `\n`. 사람이 읽기 쉽고 디버깅 용이. (필요 시 체크섬/바이너리로 확장)

### 프레임 형식
```
$<CMD>[,<arg1>[,<arg2>...]]*<CK>\n      # 체크섬 옵션(*CK 생략 가능, v1은 생략 허용)
```
- `$` 시작, `,` 인자 구분, `\n` 종료. `<CK>` = 앞 문자열 XOR(2자리 hex, 옵션).

### 호스트 → 컨트롤러 (명령)  ※ 속도 고정 → 속도 명령 없음
| 명령 | 형식 | 의미 |
|------|------|------|
| 열기 | `$OPEN` | 완전 열림까지 이동 |
| 닫기 | `$CLOSE` | 완전 닫힘까지 이동 |
| 정지 | `$STOP` | 즉시 정지 |
| 위치 | `$POS,<0-100>` | (위치 피드백 지원 시) 위치 이동 — **미정, 추후** |
| 상태조회 | `$GET` | 현재 상태 요청 |
| 핑 | `$PING` | 생존 확인 |
| 핸드셰이크 | `$HELLO,<proto_ver>` | 접속 시 프로토콜 버전 교환 |

> 1차 범위는 **OPEN/CLOSE/STOP** 만. `$POS`(위치)는 컨트롤러가 위치 피드백을 지원할 때 추가.

### 컨트롤러 → 호스트 (응답/이벤트)
| 응답 | 형식 | 의미 |
|------|------|------|
| 정상 | `#OK[,<data>]` | 명령 수락 |
| 오류 | `#ERR,<code>,<msg>` | 거부/실패 |
| 상태 | `#STA,state=<IDLE\|OPENING\|CLOSING\|STOPPED\|ERROR>,limit=<none\|open\|close>[,pos=<0-100>]` | `$GET` 응답 (pos는 피드백 지원 시) |
| 이벤트 | `#EVT,<type>[,<data>]` | 비동기 알림(끝단도달/장애물/끼임 등) |
| 핑 | `#PONG` | 응답 |

### 오류 코드(예)
`E01` 미보정, `E02` 끝단 충돌, `E03` 과전류/끼임, `E04` 모터 정지 실패, `E09` 알수없음.

### 예시 시퀀스
```
$HELLO,1   →  #OK,proto=1,fw=1.2.0
$OPEN      →  #OK
              #STA,state=OPENING,limit=none
              #STA,state=STOPPED,limit=open
              #EVT,reached_open
$GET       →  #STA,state=IDLE,limit=open
$STOP      →  #OK
```

## 설계 (보드측) — 추후 구현
- `control/motor_serial.py`: pyserial 기반 송수신 + 수신 스레드(라인 파서) + ACK 매칭(요청/응답 큐). Azone_Gateway `registrator.py`의 큐 기반 UART 구조 참고.
- `control/curtain.py`: 상태머신(`#STA` 반영) + 고수준 API(`open/close/stop`). 속도 고정이라 속도 API 없음. `set_position`은 위치 피드백 확정 후.
- 재연결: 포트 분리 감지 시 재오픈, `$PING/#PONG` 주기 헬스체크.
- 안전: ACK 타임아웃·`#ERR`·`#EVT(끼임)` 처리 → 상위에 전파, 위험 시 자동 `$STOP`.

### 안전·신뢰성 보강 (검증 반영)
- **데드맨/명령유실 대비**: `$OPEN/$CLOSE`는 끝단까지 주행하므로, 주행 중 링크 단절 시 호스트가 `$STOP`을 못 보냄.
  → **컨트롤러 측 데드맨 타이머**(호스트 하트비트 N초 없으면 자동 정지)를 규격에 포함. 물리 리밋스위치/과전류 차단이 최후 방어선.
- **`$STOP` 신뢰성**: 명령 프레임은 **체크섬 필수**(v1에서 옵션 → 필수로 격상), `$STOP`은 `#OK` 받을 때까지 재전송(멱등).
- **ACK vs 이벤트 구분**: 한 라인 스트림에 동기 ACK(`#OK/#ERR`)와 비동기(`#STA/#EVT`)가 섞임 → 명령에 **시퀀스/태그 필드** 부여하거나 "`#STA/#EVT`가 아닌 첫 `#` 응답을 ACK로" 규칙 명문화.
